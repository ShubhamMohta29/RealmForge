  [DONE]    User Auth (Login / Register / Email Confirmation)
            Supabase Auth. Session managed via middleware.
            RLS scoped to auth.uid() on every table.
            Date of birth collected at registration → stored in users.date_of_birth
            via handle_new_user trigger reading auth signup metadata.
            Email confirmation: signUp() passes emailRedirectTo using
            window.location.origin so the link works on localhost AND Vercel.
            /auth/callback route exchanges the Supabase code for a session
            and redirects to /dashboard.

  [DONE]    Content Age-Gating (PRD OQ-2)
            lib/contentRating.ts: getContentRating(dob) → under16 | teen | adult.
            DM system prompt (lib/systemPrompt.ts) injects content rules based on rating:
              under 16 — no gore, no sexual content, no profanity (child-safe)
              16–17    — vivid violence allowed, no sexual content
              18+      — no restriction
            /api/dm fetches users.date_of_birth, computes rating, passes to prompt builder.
            Existing users without DOB default to 'adult' rating.

────────────────────────────────────────────────────────────────────────────────
  DATABASE SCHEMA (Supabase / PostgreSQL)
  10 migrations applied. RLS enabled on all tables.
────────────────────────────────────────────────────────────────────────────────

  TABLE: users
    id, display_name, avatar_url, preferences, date_of_birth, created_at
    Mirrors Supabase Auth users table.
    date_of_birth added in migration 011 — drives content rating for DM prompts.

  TABLE: campaigns
    id, name, setting, dm_mode (ai|human), dm_user_id
    current_scene, world_state (JSONB), invite_code (unique 8-char)
    RLS: players see only campaigns they are members of.

  TABLE: campaign_members
    id, campaign_id, user_id
    Junction table. RLS: users see their own memberships only.

  TABLE: characters
    id, campaign_id, user_id, name, race, class, subclass
    level, xp, hp, max_hp, temp_hp, ac, speed
    ability_scores (JSONB: str/dex/con/int/wis/cha)
    skills, saving_throws (JSONB)
    death_saves (JSONB), conditions (JSONB array)
    inventory (JSONB), spells (JSONB)
    features, backstory, portrait_url, is_alive
    RLS: players see characters in their campaign; can update own.

  TABLE: messages
    id, campaign_id, character_id, user_id
    type (narration|player_action|dice_roll|combat_log)
    content, metadata (JSONB), created_at
    RLS: players see messages from their campaign only.

  TABLE: npcs
    id, campaign_id, name, description, disposition
    location, notes
    RLS: campaign members can read and update.

  TABLE: quests
    id, campaign_id, title, description
    status (active|completed|abandoned)
    xp_reward, objectives (JSONB)

  TABLE: world_memory_summaries
    id, campaign_id, summary_type (session)
    content (150-200 word summary)
    message_range_start, message_range_end, created_at

  TABLE: combat_encounters
    id, campaign_id, status (active|completed)
    round, turn_order (JSONB), current_turn_index
    monsters (JSONB), loot_table (JSONB)


────────────────────────────────────────────────────────────────────────────────
  API ROUTES
────────────────────────────────────────────────────────────────────────────────

  POST  /api/dm
        Main AI DM endpoint. Builds prompt, calls Groq 70B,
        parses XML events, applies to DB, returns narration.

  POST  /api/dm-console/narrate
        Human DM submits narration text. Applies game events
        and broadcasts to all players via Realtime.

  POST  /api/dm-console/assist
        AI Copilot (Claude Haiku) suggests what the human
        DM should say next.

  POST  /api/campaign
        Creates a new campaign with invite code. Adds creator
        as first campaign member.

  POST  /api/combat/start
        Spawns named monsters, rolls initiative for all
        combatants, creates combat_encounters record.

  POST  /api/combat/action
        Processes a player's combat turn (attack, defend,
        flee, etc.).

  POST  /api/dice
        Generic dice roll. Handles standard rolls,
        advantage/disadvantage, saving throws.

  GET   /auth/callback
        Supabase email confirmation handler. Exchanges ?code=
        param for a session, then redirects to /dashboard.
        Failure falls back to /login?error=auth_callback_failed.

────────────────────────────────────────────────────────────────────────────────
  ARCHITECTURE NOTES
────────────────────────────────────────────────────────────────────────────────

  AI PIPELINE
    Player action
      → /api/dm
      → systemPrompt.ts builds context from DB
        (campaign, characters, NPCs, quests, world memory)
      → Groq Llama 3.3 70B returns narration + embedded XML
      → gameEvents.ts parses XML tags
      → gameEventApplier.ts writes to Supabase
      → message broadcast via Supabase Realtime
      → all clients update via Zustand store

  CONTEXT MANAGEMENT
    Only the last 6 messages are sent to Groq per request.
    Every 20 narration messages, Llama 3.1 8B Instant generates
    a 150-200 word session summary stored in world_memory_summaries.
    These summaries are prepended to future prompts, giving the DM
    long-term memory without blowing the context window.

  SECURITY
    Row Level Security on every table.
    All queries scoped to auth.uid().
    Service role key only used server-side in supabaseServer.ts.
    RLS recursion on campaign_members fixed via PL/pgSQL
    function in migration 9.

  STATE MANAGEMENT
    Zustand store (gameStore.ts) holds client-side state:
    campaign, characters, messages, current encounter.
    Supabase Realtime subscriptions push server changes into
    the store. No polling — all updates are event-driven.
