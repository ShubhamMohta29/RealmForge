  SESSION 6 — 2026-05-15  (Email confirmation redirect fix)
================================================================================

  SCOPE
  ─────
  Fixed a production bug where Supabase email confirmation links redirected
  users to localhost:3000 instead of the deployed Vercel app.

  ROOT CAUSE
  ──────────
  supabase.auth.signUp() was called without an emailRedirectTo option.
  Supabase fell back to its dashboard-configured Site URL (localhost:3000),
  so confirmation links pointed there even for users on the live Vercel app.
  Additionally, there was no route handler to receive and process the
  confirmation callback — Supabase passes a one-time code via ?code= query
  param that must be exchanged server-side for a session.

  BUILT
  ─────

  1. Auth callback route — app/auth/callback/route.ts
     — GET /auth/callback: reads the ?code= param from Supabase's redirect.
     — Exchanges the code for a Supabase session using
       supabase.auth.exchangeCodeForSession(code).
     — On success → redirects to /dashboard.
     — On failure → redirects to /login?error=auth_callback_failed.
     — Uses @supabase/ssr createServerClient with async cookies() (Next.js 16).

  2. Register page — emailRedirectTo added
     — app/(auth)/register/page.tsx: signUp() options now include
       emailRedirectTo: `${window.location.origin}/auth/callback`.
     — window.location.origin is dynamic — resolves to localhost:3000 in dev
       and the real Vercel domain in production. No env var needed.

  REQUIRED MANUAL STEP (Supabase dashboard)
  ──────────────────────────────────────────
  Add allowed redirect URLs in Supabase → Authentication → URL Configuration:
    https://your-app.vercel.app/**
    http://localhost:3000/**
  Supabase validates emailRedirectTo against this list and will block unknown
  origins even if the URL is technically correct.

  1. Join campaign — dual fix
     (a) Case mismatch: handleJoin() in dashboard/page.tsx was converting the
         entered code to UPPERCASE but invite codes are lowercase UUID substrings.
         Fix: .toLowerCase() instead of .toUpperCase().
     (b) RLS block: the client-side Supabase query used the anon key, which is
         subject to RLS. The "Players can view their campaigns" policy only shows
         campaigns to existing members — a non-member querying by invite_code gets
         zero rows back even with a valid code.
         Fix: Created app/api/campaign/join/route.ts. Uses supabaseAdmin to bypass
         RLS for the invite_code lookup, upserts campaign_members, checks for an
         existing character, and returns { campaignId, dmMode, dmUserId, hasCharacter }.
         dashboard/page.tsx handleJoin() now calls this API route.

  2. Human DM — no character creation, straight to DM console
     — create-campaign/page.tsx: after campaign creation, if dmMode === 'human'
       router pushes to /campaign/[id]/dm-console instead of /create-character.
     — create-character/page.tsx: added useEffect guard. On mount, fetches the
       campaign; if dm_mode === 'human' and the user is the dm_user_id, calls
       router.replace('/campaign/[id]/dm-console') immediately.
     — dashboard/page.tsx CampaignCard: when isDM is true and campaign is not
       archived, shows ONLY the "DM Console" button (btn-amber, primary).
       The "Play →" button is hidden. The card body click also routes to dm-console.

  3. Human DM play mode — players message, DM narrates manually
     — app/api/dm/route.ts: added early-return for human DM campaigns. If
       campaign.dm_mode === 'human', saves the player_action message and returns
       { queued: true } immediately. AI (Groq) is never called.
     — play/page.tsx handleAction: skips setDMThinking(true/false) for human DM
       campaigns so the input stays live and no "DM thinking" spinner appears.
     — ActionPanel.tsx: added isHumanDM?: boolean prop.
         Human DM mode: quick action buttons hidden (they are AI prompts, not
         relevant for human DM). Placeholder changes to "Message your DM...".
         Local sending state (600ms) gives brief feedback without blocking input.
         AI DM mode: behaviour unchanged.
     — StoryLog.tsx: showThinking = isDMThinking && campaign?.dm_mode !== 'human'.
       The bouncing "Dungeon Master" dots only appear in AI mode.

  4. AI Copilot — wired + fixed
     — DMNarrationInput.tsx: added draft?: string and onDraftChange?: (text) => void
       props. Component is now fully controlled when these are provided — the AI
       Copilot's "Insert into narration →" click populates the textarea directly.
     — dm-console/page.tsx: passes draft={narrationDraft} and
       onDraftChange={setNarrationDraft} to DMNarrationInput, completing the loop.
     — AICopilot.tsx: added proper error handling — non-200 responses now surface
       the error text in the suggestion area instead of going blank.
     — app/api/dm-console/assist/route.ts:
         (a) Switched from callClaudeHaiku to callGroq. ANTHROPIC_API_KEY is not
             set in .env.local (marked "optional / future use" in .env.example),
             causing every assist call to throw silently. Groq is the active AI.
         (b) Now fetches the last 10 player_action + narration messages and includes
             them as a "Recent story log" in the system prompt so AI suggestions are
             contextually relevant to what players actually did.
         (c) Added DM auth check (dm_mode === 'human' && dm_user_id === user.id).

  5. XP level-up bug
     — Root cause: app/api/dm-console/narrate/route.ts was selecting only
       id, name, hp, max_hp, temp_hp, xp from characters. gameEventApplier.ts
       needs level, class, and ability_scores to run the level-up check
       (newLevel > c.level) and the HP increase calculation.
       With level === undefined, JavaScript evaluates (2 > undefined) as false,
       so XP was saved to the DB but the character's level never updated.
     — Fix: expanded SELECT to include class, level, ability_scores,
       proficiency_bonus, conditions, inventory.
