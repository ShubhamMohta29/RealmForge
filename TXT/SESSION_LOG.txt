  [CRITICAL — UNRESOLVED]
  After character creation, the play page opens and immediately redirects
  to /dashboard. Root cause: loadData in play/page.tsx runs a campaign
  fetch. Supabase RLS checks campaign_members to authorize the read.
  The newly-created character row triggers the redirect to /play before

  redirecting. See PENDING WORK below.

  SESSION 5 — 2026-05-14  (Content age-gating — PRD OQ-2)
================================================================================

  SCOPE
  ─────
  Implemented age-based content filtering for the AI DM, as specified in
  PRD OQ-2 (item 7.17 of the implementation plan).

  BUILT
  ─────

  1. Database migration — 011_age_gate.sql
     — Added date_of_birth date column to public.users (nullable).
     — Updated handle_new_user() trigger to read date_of_birth from
       raw_user_meta_data (passed as auth signup metadata) and insert
       it into public.users alongside display_name.

  2. Content rating utility — lib/contentRating.ts
     — getContentRating(dateOfBirth: string | null): ContentRating
     — Returns 'under16', 'teen', or 'adult' based on age computed from DOB.
     — Returns 'adult' for null/invalid DOB (backwards-compat for existing users).

  3. System prompt update — lib/systemPrompt.ts
     — contentRating added to SystemPromptContext (optional, defaults to 'adult').
     — CONTENT_RULES lookup map with per-tier rule blocks:
         under16: no gore, no sexual content, no profanity, PG combat.
         teen:    vivid violence allowed, no sexual content, mild language ok.
         adult:   no restriction block injected.
     — Content rules block inserted into the DM prompt between history and tags.

  4. DM API route update — app/api/dm/route.ts
     — Added supabaseAdmin.from('users').select('date_of_birth') to the existing
       parallel fetch (zero added latency — runs alongside campaign/character fetches).
     — Calls getContentRating() with the fetched DOB.
     — Passes contentRating to buildDMSystemPrompt().

  5. Register page update — app/(auth)/register/page.tsx
     — Added required date of birth date-picker field (type="date", max=today).
     — Helper text: "Used to set appropriate content for your age group."
     — DOB passed in supabase.auth.signUp options.data as date_of_birth.
     — Submit button disabled until all four fields (name, email, password, DOB)
       are filled.

  DESIGN DECISIONS
  ────────────────
  — DOB flows through auth metadata so the DB trigger captures it atomically
    on signup — no second API call needed, no race condition.
  — Content rules are enforced server-side in the DM prompt, not client-side —
    the client never knows the tier, and the restriction cannot be bypassed by
    manipulating the request.
  — Existing users (no DOB stored) default to 'adult' so nothing breaks.

  SESSION 6 — 2026-05-15  (Email confirmation redirect fix)
================================================================================

  SCOPE
  ─────
  Fixed a production bug where Supabase email confirmation links redirected
  users to localhost:3000 instead of the deployed Vercel app.

  ROOT CAUSE
  ──────────
  supabase.auth.signUp() was called without an emailRedirectTo option.
  Supabase fell back to its dashboard-configured Site URL (localhost:3000),
  so confirmation links pointed there even for users on the live Vercel app.
  Additionally, there was no route handler to receive and process the
  confirmation callback — Supabase passes a one-time code via ?code= query
  param that must be exchanged server-side for a session.

  BUILT
  ─────

  1. Auth callback route — app/auth/callback/route.ts
     — GET /auth/callback: reads the ?code= param from Supabase's redirect.
     — Exchanges the code for a Supabase session using
       supabase.auth.exchangeCodeForSession(code).
     — On success → redirects to /dashboard.
     — On failure → redirects to /login?error=auth_callback_failed.
     — Uses @supabase/ssr createServerClient with async cookies() (Next.js 16).

  2. Register page — emailRedirectTo added
     — app/(auth)/register/page.tsx: signUp() options now include
       emailRedirectTo: `${window.location.origin}/auth/callback`.
     — window.location.origin is dynamic — resolves to localhost:3000 in dev
       and the real Vercel domain in production. No env var needed.

  REQUIRED MANUAL STEP (Supabase dashboard)
  ──────────────────────────────────────────
  Add allowed redirect URLs in Supabase → Authentication → URL Configuration:
    https://your-app.vercel.app/**
    http://localhost:3000/**
  Supabase validates emailRedirectTo against this list and will block unknown
  origins even if the URL is technically correct.
