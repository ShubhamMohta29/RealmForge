1. AUTH PROVIDER
────────────────────────────────────────────────────────────────────────────────

  Provider:      Supabase Auth (built-in)
  Method:        Email + password (primary)
                 OAuth (Google / Discord — optional future expansion)
  Session:       JWT tokens, managed by Supabase client SDK
  Client access: Via @supabase/supabase-js (anon key, public)
  Server access: Via supabase service role key (never exposed to client —
                 used only in Next.js API routes via supabaseServer.ts)

  On new user signup:
    Supabase creates a row in auth.users.
    A database trigger (handle_new_user) fires and inserts a corresponding
    row into public.users with display_name seeded from the user's email,
    and date_of_birth read from raw_user_meta_data (set during signUp).
    This mirror pattern keeps application logic off auth.users directly.

  Environment variables required:
    NEXT_PUBLIC_SUPABASE_URL          public — used in client and server
    NEXT_PUBLIC_SUPABASE_ANON_KEY     public — used in browser client
    SUPABASE_SERVICE_ROLE_KEY         secret — server-side API routes only
    GROQ_API_KEY                      secret — AI narration (Llama 3.3 70B)
    ANTHROPIC_API_KEY                 secret — DM copilot (Claude Haiku)
    UPSTASH_REDIS_REST_URL            secret — persistent rate limiting (optional;
                                      falls back to in-memory when absent)
    UPSTASH_REDIS_REST_TOKEN          secret — Upstash Redis auth token
    NEXT_PUBLIC_SENTRY_DSN            public — Sentry error monitoring DSN
    SENTRY_ORG                        secret — Sentry org slug (for source maps)
    SENTRY_PROJECT                    secret — Sentry project slug
    SENTRY_AUTH_TOKEN                 secret — Sentry CI token (source map uploads)

2. DATABASE TABLES
────────────────────────────────────────────────────────────────────────────────

  All tables are in the public schema in PostgreSQL (Supabase).
  All tables have Row Level Security (RLS) enabled.
  Primary keys are uuid generated by gen_random_uuid().
  Timestamps are timestamptz (timezone-aware).

  ─────────────────────────────────────────────────────────────────────────────
  TABLE: public.users
  Migration: 001_users.sql + 011_age_gate.sql (adds date_of_birth)
  ─────────────────────────────────────────────────────────────────────────────

  Column          Type          Constraints
  ──────────────  ────────────  ────────────────────────────────────────────────
  id              uuid          PK, FK → auth.users(id) ON DELETE CASCADE
  display_name    text          NOT NULL (seeded with email on signup)
  avatar_url      text          nullable
  preferences     jsonb         DEFAULT '{}'
  date_of_birth   date          nullable (set from auth signup metadata)
  created_at      timestamptz   DEFAULT now()

  Notes:
    - Created automatically via trigger on_auth_user_created.
    - preferences stores UI settings (theme overrides, dice style, etc.)
    - No email column — always join to auth.users for the email.
    - date_of_birth drives content rating: under 16 (child-safe), 16–17 (teen),
      18+ (adult/unrestricted). Computed server-side via lib/contentRating.ts.
      Existing users without a DOB default to 'adult' rating.

3. FOREIGN KEY RELATIONSHIPS
────────────────────────────────────────────────────────────────────────────────

  public.users.id                  ←─ auth.users.id  (mirrors auth, CASCADE)

  campaign_members.campaign_id     →  campaigns.id  (CASCADE delete)
  campaign_members.user_id         →  users.id  (CASCADE delete)

  campaigns.dm_user_id             →  users.id  (SET NULL on delete)

  characters.campaign_id           →  campaigns.id  (CASCADE delete)
  characters.user_id               →  users.id  (CASCADE delete)

  messages.campaign_id             →  campaigns.id  (CASCADE delete)
  messages.character_id            →  characters.id  (SET NULL on delete)

  combat_encounters.campaign_id    →  campaigns.id  (CASCADE delete)

  npcs.campaign_id                 →  campaigns.id  (CASCADE delete)

  quests.campaign_id               →  campaigns.id  (CASCADE delete)
  quests.giver_npc_id              →  npcs.id  (SET NULL on delete)

  world_memory_summaries.campaign_id        →  campaigns.id  (CASCADE delete)
  world_memory_summaries.message_range_start →  messages.id  (SET NULL on delete)
  world_memory_summaries.message_range_end   →  messages.id  (SET NULL on delete)

  moderation_queue.message_id   →  messages.id  (CASCADE delete)
  moderation_queue.reported_by  →  auth.users.id  (CASCADE delete)

  CASCADE DELETE strategy:
    Deleting a campaign cascades to: campaign_members, characters, messages,
    combat_encounters, npcs, quests, world_memory_summaries.
    Deleting a user cascades to: campaign_members, characters, moderation_queue.
    Deleting a message cascades to: moderation_queue flags for that message.
    Deleting an NPC nullifies quests.giver_npc_id.

5. ROW LEVEL SECURITY — COMPLETE POLICY MAP
────────────────────────────────────────────────────────────────────────────────

  RLS is enabled on all tables. The service role key bypasses RLS —
  it is used only in server-side API routes, never on the client.

  The anon key (used in the browser) is always subject to RLS.

  TABLE: public.users
    SELECT  auth.uid() = id            (own profile only)
    UPDATE  auth.uid() = id            (own profile only)
    INSERT  via trigger only (no direct insert policy)
    DELETE  not allowed (cascade from auth.users)

  TABLE: public.campaigns
    SELECT  auth.uid() ∈ campaign_members WHERE campaign_id = id
            (must be a member to read the campaign)
    UPDATE  auth.uid() = dm_user_id    (DM only)
    INSERT  authenticated users (via API route — service role used)
    DELETE  not allowed directly

  TABLE: public.campaign_members
    SELECT  auth.uid() = user_id       (own memberships only)
    INSERT  via API route (service role)
    UPDATE  not allowed
    DELETE  not allowed directly

  TABLE: public.characters
    SELECT  auth.uid() ∈ campaign_members WHERE campaign_id = characters.campaign_id
            (any member of the campaign can see all characters)
    UPDATE  auth.uid() = user_id       (own character only)
    INSERT  auth.uid() = user_id AND user is a campaign member

  TABLE: public.messages
    SELECT  auth.uid() ∈ campaign_members WHERE campaign_id = messages.campaign_id
    INSERT  auth.uid() ∈ campaign_members WHERE campaign_id = messages.campaign_id
    UPDATE  not allowed (messages are immutable once written)
    DELETE  not allowed

  TABLE: public.combat_encounters
    SELECT  auth.uid() ∈ campaign_members (members can view combat state)
    UPDATE  via service role only (API routes control all combat mutations)
    INSERT  via service role only

  TABLE: public.npcs
    SELECT  auth.uid() ∈ campaign_members (all members see NPCs)
    INSERT  via service role only (created by AI game event parser)
    UPDATE  via service role only

  TABLE: public.quests
    SELECT  auth.uid() ∈ campaign_members
    INSERT  via service role only
    UPDATE  via service role only

  TABLE: public.world_memory_summaries
    SELECT  auth.uid() ∈ campaign_members
    INSERT  via service role only (created by summarization job)
    UPDATE  not allowed

  TABLE: public.moderation_queue
    INSERT  auth.uid() = reported_by   (users can submit their own flags)
    SELECT  auth.uid() = reported_by   (users see only their own submitted flags)
    UPDATE  not allowed directly (service role changes status to reviewed/dismissed)
    DELETE  not allowed

  SECURITY NOTE — RLS Recursion Fix:
    Early RLS policies used subqueries on campaign_members that caused infinite
    recursion (fixed in migrations 007–009). The final implementation uses a
    PL/pgSQL security definer function to check membership without recursion.

6. USER ROLES & PERMISSIONS
────────────────────────────────────────────────────────────────────────────────

  There is no formal role column on public.users. Roles are contextual,
  derived from campaign_members and campaigns.dm_user_id.

  ANONYMOUS (unauthenticated)
    Can access: / (landing), /login, /register
    Cannot access: any protected route or database table

  AUTHENTICATED PLAYER (member of a campaign, not the DM)
    Can access: /dashboard, /campaign/[id]/create-character, /campaign/[id]/play
    Cannot access: /campaign/[id]/dm-console
    DB: SELECT on campaigns, campaign_members, characters, messages, npcs,
        quests, world_memory_summaries for their campaigns.
        UPDATE on their own character row.
        INSERT messages in their campaigns.

  HUMAN DM (campaign creator, dm_mode = 'human', dm_user_id = their user id)
    Can access: all player routes PLUS /campaign/[id]/dm-console
    DB: UPDATE on the campaign row (world_state, current_scene).
        All player permissions above.
        AI copilot calls go through /api/dm-console/assist (server-side).

  AI DM CAMPAIGNS (dm_mode = 'ai', dm_user_id = null)
    No user has DM console access for AI campaigns.
    All narrative actions go through /api/dm.
    The service role key writes all game mutations (events, HP, XP, etc.)

  SERVICE ROLE (API routes only, never on client)
    Bypasses all RLS.
    Used for: all INSERT/UPDATE mutations triggered by game events,
              combat state machine, summarization jobs.

7. JSONB FIELD SCHEMAS
────────────────────────────────────────────────────────────────────────────────

  These JSONB fields store structured data. TypeScript interfaces in
  /types/character.ts enforce the shape on the application layer.

  characters.ability_scores
    { "str": 16, "dex": 14, "con": 15, "int": 10, "wis": 12, "cha": 8 }

  characters.saving_throws
    { "str": true, "dex": false, "con": true, "int": false, "wis": false, "cha": false }

    { "successes": 1, "failures": 0 }

  characters.conditions
    ["poisoned", "frightened"]

      {
        "name": "Shortsword",
        "quantity": 1,
        "weight": 2,
        "value": 10,
        "equipped": true,
        "attuned": false,
        "description": "A standard shortsword.",
        "rarity": "common"
      }

    rarity: "common" | "uncommon" | "rare" | "very_rare" | "legendary"

  characters.spells
    {
      "known": ["Fireball", "Mage Armor"],
      "prepared": ["Fireball"],
      "slots": { "1": 1, "2": 0, "3": 2 },

8. REALTIME SUBSCRIPTIONS
────────────────────────────────────────────────────────────────────────────────

  Migration: 006_realtime.sql
  All four tables are added to the supabase_realtime publication.

  Table                  Events published
  ─────────────────────  ────────────────────────────────────────────────────
  public.messages        INSERT (new narration, player actions, dice rolls)
  public.characters      UPDATE (HP, XP, conditions, inventory, death saves)
  public.combat_encounters  INSERT + UPDATE (combat start, turn advances)
  public.campaigns       UPDATE (current_scene, world_state, status changes)

  Client-side subscriptions are filtered by campaign_id using Supabase's
  channel filter syntax:
    .channel('campaign:' + campaignId)
    .on('postgres_changes', { event: 'INSERT', table: 'messages',
        filter: 'campaign_id=eq.' + campaignId }, handler)

  All clients receive updates simultaneously. There is no polling.
  The system assumes Supabase Realtime is always the source of truth for
  live state — clients do not optimistically apply updates locally.

9. API ROUTES
────────────────────────────────────────────────────────────────────────────────

  All routes are Next.js App Router route handlers (/app/api/**/route.ts).
  All routes require authentication (checked via supabaseServer.ts).
  All database mutations use the service role key (bypasses RLS).

  Method  Route                         Purpose
  ──────  ────────────────────────────  ───────────────────────────────────────
  POST    /api/dm                       AI DM narration — main game loop.
                                        Builds context → calls Groq Llama 3.3 70B
                                        → parses XML events → applies to DB
                                        → stores message → triggers Realtime.

  POST    /api/dm-console/narrate       Human DM broadcast. Stores DM narration
                                        and parses game events (same pipeline
                                        as /api/dm but no AI call).

  POST    /api/dm-console/assist        AI copilot suggestion for Human DM.
                                        Calls Claude Haiku with campaign context.
                                        Response returned only to DM, not stored.

  POST    /api/campaign/create          Create a campaign + add creator as member.
                                        Returns campaign with invite_code.

  POST    /api/campaign/join            Join campaign by invite_code.
                                        Inserts campaign_members row.
                                        (Route exists in application logic —
                                        may not have a dedicated file yet.)

  POST    /api/combat/start             Initiate combat encounter. Rolls initiative
                                        for all combatants. Creates
                                        combat_encounters row.

  POST    /api/combat/action            Resolve a player's combat action.
                                        Validates it is the player's turn.
                                        Applies attack/spell/other logic.
                                        Advances turn.

  POST    /api/dice                     Server-side dice roll. Returns full
                                        breakdown. All rolls are server-authoritative.

  POST    /api/flag                     Content moderation flag. Accepts
                                        { messageId, reason } (reason 1–500 chars).
                                        Inserts into moderation_queue. Duplicate
                                        flags from the same user are silently
                                        ignored (returns 200). Requires auth.

  GET     /auth/callback                Supabase email confirmation callback.
                                        Exchanges ?code= param for a session
                                        (supabase.auth.exchangeCodeForSession).
                                        Redirects to /dashboard on success or
                                        /login?error=auth_callback_failed on failure.
                                        emailRedirectTo in signUp() points here.

11. SENSITIVE FIELDS
────────────────────────────────────────────────────────────────────────────────

  No payment data is stored in the database. No Stripe or billing integration
  exists in the current scope.

  Sensitive environment variables (never in DB, never in client bundle):
    SUPABASE_SERVICE_ROLE_KEY   — full DB access, server-only
    GROQ_API_KEY                — AI narration, server-only
    ANTHROPIC_API_KEY           — copilot, server-only
    UPSTASH_REDIS_REST_URL      — persistent rate limiting, server-only
    UPSTASH_REDIS_REST_TOKEN    — Upstash auth, server-only
    SENTRY_AUTH_TOKEN           — source map upload, CI-only

  Player input (actions, backstory) passes through Claude/Groq as part of
  the game. All input is stripped of HTML before being sent to AI APIs
  (handled in the /api/dm route). Claude's output is rendered as plain text,
  never as HTML (no dangerouslySetInnerHTML).

  The world_state JSONB field on campaigns can contain narrative secrets
  (villain identities, hidden plot points). These are visible to all members
  via the SELECT RLS policy. If DM-only secrets are needed, they should be
  stored in npcs.notes (DM-only field) rather than world_state.

12. DATABASE TRIGGERS & FUNCTIONS
────────────────────────────────────────────────────────────────────────────────

  handle_new_user()
    Type:     AFTER INSERT trigger on auth.users
    Action:   INSERT into public.users (id, display_name, date_of_birth) on every
              new signup. date_of_birth read from new.raw_user_meta_data->>'date_of_birth'.
    Security: SECURITY DEFINER (runs with elevated privileges)
    Migration: Function updated in 011_age_gate.sql to include the DOB column.

  No other triggers currently defined. All other game state mutations are
  handled explicitly by API routes using the service role client.

  Planned (future scope):
    updated_at auto-update trigger on campaigns, characters, quests,
    combat_encounters — currently updated_at is set manually in API routes.
